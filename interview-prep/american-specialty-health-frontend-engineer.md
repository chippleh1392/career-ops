## Engineering context (so React makes sense)

### Ember → React (official ASH perspective)

- **ASHTech** (Medium, Apr 2022): [I Love Ember. Our Team Decided to Switch To React](https://medium.com/ashtech/i-love-ember-our-team-decided-to-switch-to-react-and-im-ecstatic-3eef66f01bd) — **Anthony Trama** (*“I lead the front end team at ASH”*). Summary:
  - ASH was **heavily Ember** (engines/addons) with **some Angular** in places; they **chose to consolidate on React** after a **multi-team** evaluation and **migration** planning.
  - Themes: **smaller-hiring-pool** risk with Ember, **ecosystem velocity**, **lazy loading** / **bundle** concerns, **cross-team** component sharing — classic **enterprise SPA** problems.
- **Implication for you:** A **“React front-end”** search in 2025–26 is **aligned** with a **strategic** direction, not a random keyword. You may still **touch or sit near** **legacy** stacks during migration; **ask** the team which **product** is **greenfield React** vs **migration strip**.

### Other stacks (public job text)

- Many ASH “software engineer” listings skew **.NET** (**C#, ASP.NET, SQL**, web services) — i.e. **APIs and enterprise services** you’d **integrate** with, not that you own the backend. Full-stack is **not** the stated ask for *your* front-end pipeline.

### What you’re likely building

- **SPAs** for **member, partner, and internal** users: enrollment, **gym search**, program flows, **call-center**-adjacent tools — as implied in the Medium post and in older **“Front End Engineer”**-style copy (e.g. **mobile-first**, **Scrum**, **508**). Not “marketing-only.”

## Study prep (what to review) — updated with stack research

Public signals + hydrated DOM point at **React + React Router + MUI (Emotion under the hood)** on several member surfaces; **Ember** still on at least **Active&Fit Now**-style properties. Study for the **React** track first; keep **Ember** as **migration awareness** unless the team names that product. Deeper stack notes: `interview-prep/american-specialty-health-sites-tech-stacks.md`.

### Tier 1 — must be sharp (recruiter + HM)


| Area                  | What to drill                                                                                                                                                                |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **React**             | Hooks, effects + dependency discipline, lifting state, lists/keys, performance basics (memo when it matters, not trivia).                                                    |
| **Vanilla JS**        | Arrays / objects, `async/await`, Promises, closures, modules — they say **JS fundamentals** explicitly.                                                                      |
| **MUI**               | **Layout:** `Box`, `Stack`, `Container`, **Grid** / **Grid2** (know both exist). **Inputs** and **forms**. **Theme** + `**sx`**. **Dialogs** / **AppBar** patterns.          |
| **Emotion (via MUI)** | You mostly learn this as **MUI styling**: `sx`, `styled`, theme overrides — **Emotion** is the engine that generates the `css-*` classes in DevTools, not a separate UI kit. |
| **React Router**      | Nested routes, loaders if v6.4+ (if you use them), protected routes conceptually.                                                                                            |
| **Testing**           | **Jest + React Testing Library**: query by role/label, user events, mock `fetch`.                                                                                            |


### Tier 2 — strong differentiator (Trama / ASH fit)


| Area                | What to drill                                                                                                                                                                             |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **a11y**            | **508/WCAG** mindset: focus, labels, keyboard nav, **MUI** + a11y gotchas (modals, announcements).                                                                                        |
| **API integration** | REST-shaped work: loading/error/empty, **don’t** block on C# — but know how a **frontend** consumes **versioned** HTTP APIs and handles **auth cookies** / session story at a high level. |
| **Enterprise SPA**  | **Config / feature flags** from a host (their `**$$params` + `React*`-style toggles** story): render paths by flag, don’t overfit one bundle.                                             |
| **Perf**            | **CWV** language, code splitting, **MUI** bundle awareness (import discipline).                                                                                                           |
| **Behavioral**      | 2–3 **STAR** stories: **a11y**, **DS / reusable UI**, **perf**, **cross-functional** (PM/UX), **mentoring** — Trama “hires and coaches.”                                                  |


### Tier 3 — know the exist; deep-dive if interview says so

- **.NET / backend** — You integrate with **.NET-**served **APIs**; no need to be a C# dev.
- **Third-party** — **Recurly**, **Intercom**, GTM: integration and **CSP** / layout awareness.
- **Ember** — Only if the role is on a **not-yet-migrated** product; skim **Glimmer** / route model at high level. **ASHTech** Medium posts for migration narrative.

### De-prioritize for this pipeline (unless they signal)

- **Leetcode hards**, exotic **new** frameworks, **Next.js** depth — unless the JD or HM centers them.

### Quick lab (optional)

- Skim a live **Active&Fit** or **Silver&Fit** home in DevTools: `**#root`**, **Mui***, `**css-*`** — matches what you’d ship with **MUI**.
